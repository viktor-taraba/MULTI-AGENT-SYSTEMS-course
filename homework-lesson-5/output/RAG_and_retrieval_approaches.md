# RAG (Retrieval‑Augmented Generation) — короткий довідник

Мета цього документа — чітко пояснити, що таке RAG, які етапи й ключові компоненти робочого процесу, а також описати основні підходи до retrieval (витягування інформації), їхні сильні й слабкі сторони та рекомендації з практичного застосування.

---

## 1. Що таке RAG
- RAG (Retrieval‑Augmented Generation) — архітектурний підхід для поєднання великих мовних моделей (LLM) з механізмом пошуку/витягування зовнішньої інформації під час інференсу. Замість опиратися тільки на статичні знання моделі, RAG витягує релевантні документи з зовнішніх джерел і додає їх як контекст до запиту перед генерацією відповіді.
- Основні переваги: доступ до актуальної й доменно‑специфічної інформації, зниження числа галюцинацій (якщо retrieval і фаза ранжування коректні), менша потреба у великій дозавантажувальній (re)тренуванні моделі.

## 2. Стандартний pipeline RAG
1. Індексація: підготовка корпусу — документів, баз знань, логів, технічних статей. Документи часто дроблять на чанки (chunks) певного розміру.
2. Ембеддинг / індекс: кожен документ або chunk перетворюють в індекс (sparse index або dense vectors) і зберігають у відповідному сховищі (Elasticsearch/OpenSearch для sparse; FAISS/Pinecone/Milvus/Weaviate для dense).
3. Retriever: за вхідним запитом виконується пошук — повертаються топ‑k документів/чанків.
4. (Опціонально) Reranker: переранжування знайдених документів за допомогою точнішої моделі (cross‑encoder) або late‑interaction (ColBERT‑подібні підходи).
5. Context construction (prompt stuffing): конкатенація запиту й витягнутих фрагментів у промпт для LLM (потрібно контролювати довжину і відбір контексту).
6. Generation: LLM генерує відповідь на основі комбінованого контексту.
7. (Опціонально) Attribution / citation: включення джерел та/або механізмів верифікації у відповідь.

## 3. Основні підходи до retrieval (огляд)
1. Sparse / lexical retrieval (BM25, TF‑IDF)
   - Принцип: ранжування на основі точних збігів слів (словникове представлення, багато нулів).
   - Переваги: швидкий, добре працює для точної витримки фраз, ідентифікаторів, кодів, юридичних цитат; добре підтримується Elasticsearch/OpenSearch.
   - Недоліки: не ловить семантичний зміст (синоніми, перефразування).
   - Типові бібліотеки/сервіси: Elasticsearch (BM25), Lucene, Whoosh.

2. Dense / embedding‑based retrieval
   - Принцип: репрезентація запиту й документів як щільних векторів (sentence‑transformers, OpenAI embeddings etc.) і пошук найближчих сусідів у векторному просторі.
   - Переваги: дозволяє знайти документи з тією самою ідеєю, навіть без точних збігів слів; краще для семантичних запитів.
   - Недоліки: дорожчий в індексації/запитах; вразливий до шуму; потребує ANN (approximate nearest neighbor) для масштабності; шанс повернути некоректно інтерпретований контекст.
   - Типові движки: FAISS, Annoy, HNSWlib; продукти: Pinecone, Milvus, Weaviate, Qdrant.

3. Hybrid retrieval (sparse + dense)
   - Принцип: поєднання сильних сторін BM25 (точні збіги) і vector search (семантика), або послідовне/паралельне злиття результатів.
   - Переваги: краща точність у широкому спектрі запитів; часто практично найкращий вибір у продукції.
   - Варіанти: ранжування через сумарні скорінги, розширення запиту (query expansion), late interaction (ColBERT/ColBERTv2).

4. Neural retrievers / dense bi‑encoder vs cross‑encoder
   - Bi‑encoder: окремо ембеддять запит і документи, швидкий варіант (дозволяє ANN). Добрий для retrieval step.
   - Cross‑encoder: подає запит + документ разом у модель для точного скорінгу — набагато точніший, але дорожчий; зазвичай використовується як reranker для топ‑N від bi‑encoder.

5. Late interaction та ColBERT‑подібні підходи
   - Поєднують ефективність векторного пошуку з детальнішою взаємодією між термінами запиту й документу після попереднього відбору; дають покращене ранжування без повного використання cross‑encoder на всьому корпусі.

6. Knowledge Graph / Graph‑based retrieval
   - Використовують структуру знань (суб’єкт‑предикат‑об’єкт) для відповіді на запити, особливо корисно для зв’язних фактів, бізнес‑логіки або коли важлива інтеграція онтологій.

## 4. Практичні компоненти та інструменти
- Chunking: розбивка документів (часто 200–1000 токенів) — впливає на точність і контекстність.
- Metadata filtering: фільтрація за датою, автором, джерелом — підвищує релевантність.
- Vector DB: FAISS (локально), HNSWlib, Pinecone, Milvus, Qdrant, Weaviate.
- Sparse search: Elasticsearch / OpenSearch (BM25).
- Reranker: Hugging Face cross‑encoders, T5‑ranker, BERT/RoBERTa cross‑encoders.
- Retrieval frameworks: LlamaIndex, Haystack, LangChain (механіки для комбінування retrieval + LLM).

## 5. Метрики оцінки retrieval
- Recall@k (важлива для RAG — якщо релевантних документів немає в топ‑k, LLM не зможе використати їх)
- MRR (Mean Reciprocal Rank)
- nDCG (normalized Discounted Cumulative Gain)
- Precision@k
- Human evaluation для кінцевої генерації (фактова коректність відповіді)

## 6. Коли обрати який підхід — рекомендації
- BM25 (sparse): коли потрібні точні збіги (коди, назви, цитати), або обмежений бюджет / велика частота простих запитів.
- Dense embeddings: коли запити можуть бути перефразовані, потрібно семантичне розуміння (FAQ‑matching, семантичний пошук в документах).
- Hybrid: загальний рекомендаційний вибір для продукційних RAG — поєднує точність і семантику.
- Graph‑based: коли дані мають явні звязки, або потрібні складні ланцюги логіки.

## 7. Best practices та практичні ради
- Контролюйте розмір контексту: не перегружайте LLM великою кількістю низькоякісних фрагментів.
- Використовуйте reranker для покращення точності перед генерацією (особливо якщо corpus великий).
- Включайте джерела в відповіді (citations) — це підвищує довіру та допомагає верифікації.
- Регулярно оновлюйте індекс та відслідковуйте датованість даних.
- Тестуйте retrieval під прикладами реальних запитів користувачів (hard negative sampling при тренуванні retriever).
- Для масштабування використовуйте ANN (HNSW, IVF+PQ в FAISS) і моніторьте latency vs accuracy.

## 8. Падіння та обмеження RAG
- Наявність релевантних документів у топ‑k є критичною — поганий retrieval = поганий результат, незалежно від сили LLM.
- LLM може неправильно інтерпретувати контекст (витягнуті фрагменти), що призведе до помилкових висновків.
- Питання гарантій правдивості: RAG допомагає, але не усуває потребу у валідації фактів.

## 9. Коротка архітектура (приклад)
- Ingest (PDF/DB/HTML) → Chunker → Embedder → Vector DB + Metadata store
- Query → Retriever (hybrid: BM25 + vector top‑k) → Reranker (cross‑encoder) → Context builder → LLM → Response + citations
